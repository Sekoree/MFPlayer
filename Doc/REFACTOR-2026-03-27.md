# S.Media Refactor Plan — 2026-03-27

> Single-document plan for the consolidation, simplification and cleanup pass agreed on 2026-03-27.
> Supersedes per-step guidance in `SIMPLIFICATION-PLAN.md` (which tracks the *previous* 7-step work).

## Goals

1. **Unify the mixer hierarchy** — collapse standalone `AudioMixer`, `VideoMixer` and `AudioVideoMixerRuntime` into a single many-to-many `AudioVideoMixer` that "just works".
2. **Wire `MediaPlayer.Play()` end-to-end** — consumers should never need to call `StartPlayback` separately.
3. **Simplify drift correction** — flatten to a minimal proportional + hard-resync model; remove the `PlaybackClockAnchor` indirection.
4. **Remove placeholder / legacy leftovers** — kill `placeholder_rgba` / `pcm_f32le` stubs in `FFSharedDecodeContext` and `FFMediaItem`, prune legacy packages from `Directory.Packages.props`, and remove `VideoLibs` legacy projects from the solution.
5. **Collapse `VideoPresenterSyncPolicy`** — merge Hybrid/StrictAv into one parameterised path.
6. **Fix or retire broken test-app wiring** — `AudioMixerTest` and `VideoMixerTest` currently bypass the mixer runtime; update them to use the unified `AudioVideoMixer` or delete them.

### Step 1b — Absorb runtime into `AudioVideoMixer`

The current `AudioVideoMixerRuntime` pumps the **first** audio source to the **first** audio output and the **first** video source to the **first** video output. This is too restrictive. The new `AudioVideoMixer` will:

* **Audio pump thread** — for each pump cycle, read from *every* started `IAudioSource`, mix into a shared buffer (additive float mix, clamped), then push the mixed frame to *every* registered `IAudioOutput`. Sample-accurate start-offsets per source are respected.
* **Video read thread** — read from the **active** video source (selected via `SetActiveVideoSource`), enqueue into a presentation queue.
* **Video presentation** — either on a background thread or caller-ticked (`PresentOnCallerThread`), dequeue from the presentation queue using `VideoPresenterSyncPolicy` and push to *every* registered `IVideoOutput`.
* **Drift correction** — simplified (see Step 3), driven from the audio pump.

The `AudioVideoMixer` will own these threads directly (no separate `Runtime` object). `StartPlayback(AudioVideoMixerConfig)` starts the threads. `StopPlayback()` tears them down.

## Step 3 — Simplify drift correction

### Current state (being removed)

* `PlaybackClockAnchor` — Stopwatch-based timeline interpolator used by the video presenter.
* `DriftCorrectionState` — 6-param gain/deadband/hard-resync model with `0.995` decay and `*8` hard-resync multiplier.
* Audio pump calls `mixer.Seek(correctedTimelineSeconds)` every cycle, video presenter calls `PlaybackClockAnchor.Estimate()`.

### New model

**Remove `PlaybackClockAnchor` entirely.** The audio pump already keeps `IMediaClock` updated via `Seek()`; the video presenter will read `IMediaClock.CurrentSeconds` directly.

**Replace `DriftCorrectionState` with `SimpleDriftCorrection`:**

```
Parameters (internal, not consumer-facing):
  DeadbandMs     = 10      — ignore drift below this
  Gain           = 0.15    — proportional correction
  MaxStepMs      = 5       — clamp per-tick correction
  MaxOffsetMs    = 200     — clamp accumulated offset
  HardResyncMs   = 300     — if abs(drift) exceeds this, jump directly

Logic (once per second):
  if abs(drift) < DeadbandMs → decay offset towards 0
  elif abs(drift) >= HardResyncMs → reset offset to -drift * 0.5, increment resync counter
  else → offset += clamp(drift * Gain, -MaxStepMs, MaxStepMs); offset = clamp(offset, -MaxOffsetMs, MaxOffsetMs)
```

This is essentially the same algorithm minus the tunable consumer parameters. The 6 drift params stay internal with sensible defaults. `AudioVideoMixerConfig` does **not** expose them.

### `AudioVideoMixerDebugInfo` — no changes

The debug snapshot is well-designed. We keep all fields; the simplified correction still produces the same metrics.

## Step 5 — Collapse `VideoPresenterSyncPolicy` Hybrid/StrictAv paths

### Current state

`Hybrid` and `StrictAv` modes share identical branching logic — the only difference is the max-wait constant (`2 ms` vs `3 ms`). 

### New model

Keep **two** modes:
* `Realtime` (renamed from `Stable`) — always present freshest frame, coalesce older.
* `Synced` (replaces both `Hybrid` and `StrictAv`) — clock-aligned presentation with a single configurable `MaxWait` (default 2 ms, consumer can set higher for stricter sync).

`AudioVideoSyncMode` enum:
```csharp
public enum AudioVideoSyncMode
{
    Realtime = 0,   // was Stable — present freshest, coalesce
    Synced = 1,     // was Hybrid/StrictAv — clock-aligned
}
```

`VideoPresenterSyncPolicyOptions` drops `HybridMaxWait` / `StrictMaxWait`, gets a single `MaxWait`.

### Migration

All call-sites using `Stable` → `Realtime`, `Hybrid`/`StrictAv` → `Synced`. Test apps' `--sync-mode` flags updated.
