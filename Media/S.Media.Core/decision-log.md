# S.Media Core Decision Log

Tracks implementation-impacting decisions that refine (but do not replace) `Media/S.Media.Core/PLAN.smedia-architecture.md` and `Media/S.Media.Core/API-outline.md`.

## DEC-2026-03-24-001: Source Identity Is Implementation-Generated

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` source contracts and mixer remove-by-id behavior
- Decision:
  - `IAudioSource.SourceId` and `IVideoSource.SourceId` are generated by the source implementation.
  - `SourceId` is immutable and stable for the source instance lifetime.
- Consequences:
  - `RemoveSource(Guid sourceId)` resolves against `SourceId` only.
  - `RemoveSource(instance)` and `RemoveSource(Guid)` must target the same registration semantics.

## DEC-2026-03-24-002: VideoFrame Is Pooled and Caller-Owned

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` video source/output frame ownership contracts
- Decision:
  - `VideoFrame` instances are pool-backed and caller-owned.
  - Caller must dispose every acquired `VideoFrame`.
  - Implementations may reuse internal decode/convert memory, but frame contents remain valid until `VideoFrame.Dispose()`.
- Consequences:
  - Output paths (`IVideoOutput`) must not assume ownership transfer.
  - `Dispose()` must be safe and deterministic for pooled return semantics.

## DEC-2026-03-24-003: Canonical Player Signature

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: Core player contract wording
- Decision:
  - Canonical API is `int Play(IMediaItem media)`.
  - `Play(Media)` in prose/docs is shorthand only and not a distinct signature.
- Consequences:
  - All module and architecture docs use the canonical signature in API sections.
  - Any shorthand mention must explicitly point to `IMediaItem`.

## DEC-2026-03-24-004: Transitional Coexistence, Hard-Cut at Sign-Off

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: repo migration policy
- Decision:
  - Legacy `Seko.OwnAudioNET.*` and new `S.Media.*` may coexist temporarily during migration.
  - Final runtime sign-off requires full removal of legacy runtime paths/projects from the solution graph.
  - Compatibility wrappers/shims remain disallowed in the target architecture.
- Consequences:
  - Migration work can proceed incrementally, but completion criteria include legacy removal and dependency cleanup (`OwnAudio`).

## DEC-2026-03-24-005: VideoFrame Dispose Contract

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` frame lifetime and pool return safety
- Decision:
  - `VideoFrame.Dispose()` is idempotent.
  - `VideoFrame.Dispose()` is cross-thread safe for producer/consumer decoding paths.
- Consequences:
  - Repeated dispose calls are no-op after first successful release.
  - Pool return/release implementation must be safe when dispose is invoked from a different thread than decode/read.

## DEC-2026-03-24-006: SourceId Uniqueness Scope

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` source identity semantics
- Decision:
  - `SourceId` remains implementation-generated and immutable per instance.
  - `SourceId` should be unique per process lifetime when feasible.
  - Minimum guarantee: no collisions among live/registered source instances.
- Consequences:
  - Mixer remove-by-guid operations remain deterministic under concurrent multi-source usage.
  - Implementations should avoid ID reuse strategies that can collide during process lifetime.
  - Duplicate `SourceId` registration in mixer source collections must be rejected with a deterministic owned error code (`MixerSourceIdCollision`, `3001`).

## DEC-2026-03-24-007: Pixel Format Data Model

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` video frame format contracts
- Decision:
  - Use `IPixelFormatData` as the format metadata contract for `VideoFrame`.
  - Supported formats expose dedicated typed structs implementing `IPixelFormatData`.
- Consequences:
  - Core keeps a shared format enum for high-level switching, but per-format fields are owned by typed data structs.
  - Backends/adapters can validate format-specific plane/stride invariants deterministically.

## DEC-2026-03-24-008: Configurable Video Output Backpressure

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` output push semantics
- Decision:
  - Video output backpressure behavior is configurable.
  - Backpressure failure paths use deterministic owned error codes.
- Consequences:
  - Queue-full/no-progress conditions are observable with stable return codes (`VideoOutputBackpressureQueueFull`, `4000`; `VideoOutputBackpressureTimeout`, `4001`).
  - Policy choice is explicit in output configuration and not hidden in backend defaults.

## DEC-2026-03-24-009: Video Output Backpressure Defaults and Disposed-Frame Rejection

- Date (UTC): `2026-03-24`
- Status: `Accepted`
- Scope: `S.Media.Core` video output runtime defaults and push validation
- Decision:
  - Default `VideoOutputBackpressureMode` is `DropOldest`.
  - Wait-mode timeout default is derived from effective frame duration multiplied by a configurable frame-time multiplier.
  - `PushFrame(...)` must reject disposed frame instances with deterministic code `VideoFrameDisposed` (`4002`).
- Consequences:
  - Default runtime behavior favors continuity/low-latency rendering over strict queue preservation.
  - Wait-mode timeout remains cadence-aware while allowing explicit override when cadence is unresolved.
  - Disposed-frame misuse is observable via stable non-zero return code instead of undefined behavior.
